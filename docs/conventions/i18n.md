# Translations & i18n conventions

The frontend uses [`i18next`](https://www.i18next.com) configured in
`frontend/src/i18n/translations.tsx`. Resources live in
`frontend/src/i18n/en.json`. Non-English locales fall back to English via
i18next's `fallbackLng`.

## Hook: always use `useFixedT(prefix)`

```ts
import { useFixedT } from "~shared/hooks/useFixedT";

const tDetails = useFixedT("organization:details");
return <h2>{tDetails("general:title")}</h2>;  // "General"
```

Don't call `useTranslation()` directly — `useFixedT` is the thin wrapper
(`src/shared/hooks/useFixedT.ts`) and consistent across the codebase. It
takes a `keyPrefix` so the returned `t(...)` resolves relative paths.

## Key path separator is `:` (not `.`)

`i18next` is configured with `keySeparator: ":"`. Nested JSON objects map to
colon-separated key paths:

```json
{ "organization": { "details": { "general": { "title": "General" } } } }
```
→ `t("organization:details:general:title")` → `"General"`.

The legacy `nsSeparator: ";"` is set but unused — we only use the default
namespace (`"mpt"`).

## Top-level namespaces

| Namespace | Purpose | Examples |
|---|---|---|
| `<feature>` | Strings owned by one feature. Add a new top-level key only when adding a new feature. | `organization:*`, `entitlement:*` |
| `shared` | Cross-feature strings: navigation, grid columns/fields, generic actions, properties shared by multiple entities. | `shared:nav:organizations`, `shared:grid:columns:status`, `shared:properties:linkedDataSource` |

## Nesting convention

Inside a namespace, group by concern. The patterns currently in use:

- `<feature>:details:<section>:title` — the label shown in tabs / breadcrumbs
  (e.g. `organization:details:general:title`).
- `<feature>:details:<section>:properties:<field>` — labels for the
  property rows on a detail page.
- `<feature>:errors:<reason>` — failure messages, often suffixed with an HTTP
  status (see *Dynamic keys*).
- `<feature>:placeholders:<field>` — form field placeholders.
- `<feature>:create_info`, `<feature>:add_<entity>`, `<feature>:save`,
  `<feature>:cancel` — modal/form action strings.
- `shared:grid:columns:<field>` — grid column headers.
- `shared:grid:fields:<field>` — filter field labels (read by RQL grid
  config — distinct from `columns`, which is the header text).
- `shared:grid:views:<view>` — named grid views.
- `shared:grid:dataSourceType:<type>` — labels for datasource type values
  (`azure_cnr`, `aws_cnr`, `gcp_cnr` — see *Dynamic keys*).
- `shared:nav:<route>` — top-level navigation labels.
- `shared:properties:<field>` — property labels that appear in more than one
  feature's detail view.
- `shared:placeholders:<field>` — generic placeholders.
- `shared:details:additionalIds`, `shared:details:references` — section
  headers on detail pages.
- `shared:actions:<verb>` — generic action labels (currently only `cancel`).

## Leaf key naming

The codebase is mixed and that's intentional:

- **`snake_case`** when the key mirrors an API field name. This keeps the
  key identical to what the backend sends — e.g. `affiliate_external_id`,
  `parent_id`, `expenses_so_far_this_month`, `display_name`, `data_source`,
  `azure_cnr`. Tooling that maps API responses to labels (grids, property
  lists) can use the raw API field as the lookup key without conversion.
- **`camelCase`** for UI-only labels with no direct backend tie —
  `lastLogin`, `createdAt`, `billingCurrency`, `rolesCount`, `forecastThisMonth`,
  `contactEmail`.
- **`snake_case` with verbs** for compound action keys — `add_user`,
  `add_entitlement`, `create_admin_info`, `create_failed`.

Rule of thumb: **if the key is derived from an API field, mirror the field
exactly; otherwise camelCase**. Don't translate snake_case API fields into
camelCase keys — you'll break the dynamic-lookup sites.

## Dynamic keys (template lookups)

Some call sites compute the key at runtime:

```ts
tErrors("add_admin_failed_with_code_" + (err.status || "unknown"));
tDataSourceType(item.type);                  // → "azure_cnr" | "aws_cnr" | ...
tProperties(field);                          // → "created" | "updated" | ...
```

When you add a dynamic call site, **enumerate every possible suffix you'll
look up and add a key for each**. Today:

- HTTP-status errors at `organization:users:errors:add_admin_failed_with_code_<status>`
  cover `400 / 403 / 404 / 409 / 422 / 500 / 502 / unknown`. Other statuses
  (e.g. `401`, `429`) silently fall through to the `unknown` fallback.
- Datasource types at `shared:grid:dataSourceType:*` cover `azure_cnr`,
  `aws_cnr`, `gcp_cnr`. Other types render the raw key.
- Event names at `shared:properties:*` cover `created`, `updated`,
  `redeemed`, `terminated`.

Document the closed set of suffixes in the JSON; future readers can grep
for the prefix and find everything covered.

## Adding a new locale

`translations.tsx` currently does a single static `import("./en.json")`.
For a new locale:

1. Add `frontend/src/i18n/<lang>.json` (matching the key tree of `en.json`,
   keys that don't translate can fall back via i18next).
2. Replace the static import with a map keyed on `language`:
   ```ts
   const loaders: Record<string, () => Promise<unknown>> = {
     en: () => import("./en.json"),
     fr: () => import("./fr.json"),
   };
   const translations = (await loaders[language]?.()).default ?? {};
   ```
3. Make sure `languageCodes` from `@swo/countries` accepts the new code.

Don't reintroduce the `import(\`./${locale}/${locale}.json\`)` template
import — esbuild's dynamic-glob expansion was the source of the historical
`empty-glob` build warning.

## Tooling

- `frontend/src/i18n/sortTranslations.ts` sorts keys alphabetically within
  each object. Run it after editing `en.json` to keep diffs minimal.
